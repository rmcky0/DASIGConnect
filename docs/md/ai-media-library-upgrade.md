## 1. Root Problem Summary

Your current pipeline stores only **one text-based embedding** per asset. Accuracy depends entirely on what Claude described at upload time.

**Current broken chain:**

```
Cookie uploaded
→ Claude forced into academic categories: "Innovation, Technology, Research"
→ Voyage embeds wrong text description
→ Vector lands in "academic/tech" neighborhood
→ Food searches never find it
→ Suggestions return wrong assets
```

**Fixed chain:**

```
Cookie uploaded
→ Claude correctly says: "chocolate cookie, red velvet dessert, close-up food photo"
→ Voyage embeds correct description
→ Vector lands in "food/product photography" neighborhood
→ Suggestions work correctly
```

**Two things are broken — both must be fixed:**

| Problem | Root Cause | Fix |
|---|---|---|
| Wrong tags on food/product photos | Claude prompt biased toward academic categories | Rewrite classification prompt |
| Old assets have wrong vectors | Bad embeddings already stored in DB | Reclassification job |
| Single embedding limits accuracy | Only text/semantic vector stored | Add image embedding per asset |

## 2. Fix Claude Classification Prompt

This is the **highest priority fix**. Everything downstream depends on it.

### Current Problem

Claude is forced into a narrow set of institutional categories:
- `Workshop`, `General Event`, `Research`, `Innovation`, `Technology`, `Students`

This causes food photos, product shots, and portraits to be misclassified.

### New Prompt

Use this at upload time:

```
Analyze this media asset for a university media library.

Return ONLY valid JSON. No explanation. No markdown.

Your job is to identify what is ACTUALLY VISIBLE in this image.
Do NOT force it into academic, research, innovation, technology, student,
or event categories unless there is clear visual evidence.

Allowed primary_category values:
[
  "Food",
  "People",
  "Event",
  "Technology",
  "Research",
  "Education",
  "Sports",
  "Culture",
  "Nature",
  "Document",
  "Product",
  "Architecture",
  "Artwork",
  "Other"
]
Asset Type
[
  "Product Photo",
  "Food Photo",
  "Event Photo",
  "Lab Photo",
  "Project Presentation",
  "Poster",
  "Document",
  "Screenshot",
  "Portrait",
  "Group Photo",
  "Landscape",
  "Building Photo",
  "Artwork Photo",
  "Infographic",
  "Other"
]

Return this exact structure:
{
  "primary_category": "",
  "asset_type": "",
  "ai_caption": "",
  "visible_objects": [],
  "specific_subjects": [],
  "visual_style": [],
  "dominant_colors": [],
  "possible_use_cases": [],
  "ai_tags": [],
  "excluded_categories": [],
  "confidence": 0.0
}
```

`ai_caption` is the generated caption/description field in Claude's JSON response. During implementation, store it in the existing `media_assets.ai_description` column to avoid unnecessary schema churn. Add a separate `ai_caption` column only if the entity, DTOs, frontend, and migration plan are updated together.

`ai_tags` is the normalized tag list generated by Claude. Store canonical tags in `asset_tags` with `source = 'ai_generated'`. Optionally also cache the same list in `media_assets.ai_tags` (`TEXT[]`) for faster reads.

`primary_category` is the new prompt field. The current implementation already has `media_assets.ai_category`; keep writing `primary_category` into `ai_category` during the compatibility phase, or explicitly migrate to a new `primary_category` column after all reads are updated. Do not leave both fields populated with divergent values.

## 3. Database Schema Changes

### Add metadata fields to `media_assets`

```sql
ALTER TABLE media_assets ADD COLUMN asset_type            TEXT;
ALTER TABLE media_assets ADD COLUMN visible_objects       TEXT[];
ALTER TABLE media_assets ADD COLUMN specific_subjects     TEXT[];
ALTER TABLE media_assets ADD COLUMN visual_style          TEXT[];
ALTER TABLE media_assets ADD COLUMN dominant_colors       TEXT[];
ALTER TABLE media_assets ADD COLUMN possible_use_cases    TEXT[];
ALTER TABLE media_assets ADD COLUMN ai_tags               TEXT[]; -- optional cached read model; canonical rows still live in asset_tags
ALTER TABLE media_assets ADD COLUMN excluded_categories   TEXT[];
ALTER TABLE media_assets ADD COLUMN ai_confidence         NUMERIC;
ALTER TABLE media_assets ADD COLUMN reclassified_at       TIMESTAMP;
ALTER TABLE media_assets ADD COLUMN status                TEXT;
```

`media_assets.ai_description` already exists in the current branch. Keep it as the DB storage for Claude's `ai_caption` response field unless there is a separate, deliberate rename migration.

`primary_category` from Claude should map into the existing `media_assets.ai_category` column during the compatibility phase. Do not add a separate `primary_category` column unless all repository, DTO, and frontend reads are migrated at the same time.

`media_assets.deleted_at` already exists in the current schema as `TIMESTAMPTZ`; do not re-add it or change it to `TIMESTAMP`.

Add `status` as nullable first, then backfill before setting the default. Do not add it with `DEFAULT 'PROCESSING'` up front, because existing embedded assets would become invisible to `status = 'READY'` suggestion queries.

```sql
ALTER TABLE media_assets
    ADD COLUMN IF NOT EXISTS status TEXT;

UPDATE media_assets
SET status = 'READY'
WHERE deleted_at IS NULL
  AND embedding IS NOT NULL
  AND status IS NULL;

UPDATE media_assets
SET status = 'DELETED'
WHERE deleted_at IS NOT NULL
  AND status IS NULL;

UPDATE media_assets
SET status = 'PROCESSING'
WHERE status IS NULL;

ALTER TABLE media_assets
    ALTER COLUMN status SET DEFAULT 'PROCESSING';

ALTER TABLE media_assets
    ADD CONSTRAINT chk_media_assets_status
        CHECK (status IN ('PROCESSING', 'READY', 'FAILED', 'DELETED'));
```

New uploads should start as `PROCESSING`, then switch to `READY` only after required metadata and embeddings are stored.

### Create `media_asset_embeddings` table

Replace the single `media_assets.embedding` column with a dedicated table:

```sql
CREATE TABLE media_asset_embeddings (
    id               UUID PRIMARY KEY DEFAULT gen_random_uuid(),
    asset_id         UUID NOT NULL REFERENCES media_assets(id) ON DELETE CASCADE,
    embedding_type   TEXT NOT NULL CHECK (embedding_type IN ('image', 'semantic', 'keyframe')),
    embedding        VECTOR(1024) NOT NULL,
    model            TEXT NOT NULL,
    created_at       TIMESTAMP DEFAULT now()
);

-- HNSW index for fast approximate nearest-neighbor search
CREATE INDEX ON media_asset_embeddings
    USING hnsw (embedding vector_cosine_ops)
    WITH (m = 16, ef_construction = 64);

-- Index for filtering by type
CREATE INDEX ON media_asset_embeddings (asset_id, embedding_type);

-- Prevent duplicate vectors after upload retries or reclassification.
CREATE UNIQUE INDEX IF NOT EXISTS uq_media_asset_embedding_type
    ON media_asset_embeddings(asset_id, embedding_type);
```

Enable RLS on `media_asset_embeddings` and scope it through the parent `media_assets` row:

```sql
ALTER TABLE media_asset_embeddings ENABLE ROW LEVEL SECURITY;

CREATE POLICY media_asset_embeddings_institution_isolation
    ON media_asset_embeddings
    USING (
        EXISTS (
            SELECT 1
            FROM media_assets ma
            WHERE ma.id = media_asset_embeddings.asset_id
              AND ma.deleted_at IS NULL
              AND (
                  ma.institution_id = nullif(current_setting('app.current_institution_id', true), '')::UUID
                  OR current_setting('app.current_role', true) = 'administrator'
              )
        )
    );
```

> **Why HNSW?** pgvector supports both HNSW and IVFFlat indexes. HNSW gives better recall at query time and is recommended for most use cases.

## 4. Store Dual Embeddings

### Why Two Embeddings?

| Embedding Type | What It Captures | Example for Cookie |
|---|---|---|
| **Image embedding** | Visual similarity — shapes, colors, composition | Dessert shape, red velvet color, cream topping, dark plate |
| **Semantic embedding** | Meaning, context, use case | Food, dessert, bakery, product photo |

Using only one leaves gaps. Together they give much stronger retrieval.

### How to Generate Both

```
Asset uploaded
↓
Claude Vision → structured JSON metadata
↓
Voyage multimodal:
  ├── multimodal image embedding(image input) → image_embedding    (embedding_type: "image")
  └── text embedding(claude_text_blob)        → semantic_embedding (embedding_type: "semantic")
↓
Store both rows in media_asset_embeddings
```

Use Voyage multimodal embedding for image input. Use Voyage text embedding for semantic text input. Do not send raw image bytes to the existing text-only embedding method.

Implementation notes:
- Image/document vectors: use Voyage multimodal embeddings, `POST /v1/multimodalembeddings`, model `voyage-multimodal-3.5`, `input_type = "document"`.
- Image query vectors: use the same multimodal endpoint/model, `input_type = "query"`.
- Semantic asset vectors: use Voyage text embeddings, `POST /v1/embeddings`, keep the configured text model at 1024 dimensions.
- Semantic query vectors: use the text embedding endpoint with `input_type = "query"`.
- Keep all stored vectors at 1024 dimensions unless a migration explicitly changes every vector column and index together.

### Voyage Embedding Text Blob (semantic)

```
asset_code: ASSET-DB3B
filename: red-velvet-cookie.jpg
media_type: IMAGE
uploaded_month: 2026-05
primary_category: Food
asset_type: Product Photo
caption: A close-up product photo of a red velvet dessert with cream topping on a dark plate.
visible_objects: dessert, cream, plate, white drizzle
ai_tags: food, dessert, red velvet, cream, bakery, product photo
dominant_colors: red, white, black
possible_use_cases: bakery menu, food promotion, product showcase
```

## 5. Updated Upload Flow

```
User picks file in browser
↓
Frontend uploads binary to Supabase Storage
↓
Frontend sends metadata to backend:
  POST /api/v1/media-assets
  { storageUrl, fileName, fileType, fileSizeBytes }
↓
Backend saves MediaAsset row (basic profile)
↓ (async — upload response already returned)

AIClassificationService.classifyAndEmbed(assetId, storageUrl)
│
├── Step 1: Fetch image bytes → base64 encode
│
├── Step 2: Claude Vision (new strict prompt)
│     Returns: primary_category, asset_type, ai_caption,
│              visible_objects, specific_subjects, visual_style,
│              dominant_colors, possible_use_cases,
│              ai_tags, excluded_categories, confidence
│
├── Step 3: Save new metadata fields to media_assets row
│
├── Step 4: Save tags to asset_tags (source: "ai_generated")
│
├── Step 5: Generate IMAGE embedding
│     Voyage multimodal image embedding(image input) → 1024-dim vector
│     INSERT INTO media_asset_embeddings (embedding_type: "image")
│
└── Step 6: Generate SEMANTIC embedding
      Build text blob from Claude output
      Voyage text embedding(textBlob) → 1024-dim vector
      INSERT INTO media_asset_embeddings (embedding_type: "semantic")

When both metadata and required embeddings are saved, set `media_assets.status = 'READY'`. If classification or embedding fails, leave the asset out of suggestions until reconciliation succeeds.
```

## 6. Updated AI Suggestion Flow

### Option A — Text-Based Query (existing submissions flow)

No change needed here. This still works for event title + caption + tags queries.

### Option B — Image-Based Query (new: user uploads temp image)

```
User uploads temporary query image
↓
POST /api/v1/ai/suggestions/by-image
{ tempImageUrl or base64 }
↓
AIRecommendationService.suggestByImage()
│
├── Step 1: Claude analyzes query image ONCE
│     Returns: primary_category, visible_objects, excluded_categories, etc.
│
├── Step 2: Generate query embeddings
│     Voyage multimodal image embedding(query image input) → image_query_vector
│     Voyage text embedding(claudeQueryText)               → semantic_query_vector
│
├── Step 3: pgvector search — IMAGE embeddings
│     SELECT asset_id, 1-(embedding <=> imageQueryVector) AS image_score
│     FROM media_asset_embeddings
│     WHERE embedding_type = 'image'
│       AND asset_id IN (
│           SELECT id FROM media_assets
│           WHERE institution_id = ?         -- permission filter
│             AND status = 'READY'
│             AND deleted_at IS NULL
│             AND (role logic here)
│       )
│     ORDER BY embedding <=> imageQueryVector
│     LIMIT 30
│
├── Step 4: pgvector search — SEMANTIC embeddings
│     Same query using semantic_query_vector
│     LIMIT 30
│
├── Step 5: Merge + deduplicate candidates
│     Combine both result sets (~40–50 unique assets)
│
├── Step 6: Score each candidate
│     (See scoring formula below)
│
├── Step 7: Optional Claude reranking (top 10–20 only)
│     Send query image + top candidates metadata to Claude
│     Claude returns ranked list with reasons
│
└── Step 8: Return top 6–12 suggestions with matchReasons
```

### Permission Filter (applied before vector search)

```sql
WHERE (
    :role = 'ADMINISTRATOR'
    OR (:role = 'VALIDATOR' AND ma.institution_id = :institution_id)
    OR (:role = 'CONTRIBUTOR' AND ma.uploaded_by_user_id = :user_id)
)
AND ma.status = 'READY'
AND ma.deleted_at IS NULL
```

Use the Java role names at the application boundary (`ADMINISTRATOR`, `VALIDATOR`, `CONTRIBUTOR`). If the SQL depends on the request-scoped database setting, use the existing lowercase value (`administrator`) consistently with the tenant interceptor/RLS policies.

### Optional Claude Rerank Prompt

Send only the query image + top 10–20 candidates (metadata/thumbnails). Never the full library.

```
You are reranking media library suggestions.

The user uploaded the query image. Select the assets most relevant
for reuse in content submission.

Prioritize:
1. Same main subject
2. Same content type or use case
3. Same visual context and category
4. Similar objects, colors, and composition

Do NOT rank based only on color similarity.
Reject assets that are only superficially similar.

Return JSON only:
{
  "ranked_assets": [
    {
      "asset_id": "",
      "rank": 1,
      "match_score": 0.0,
      "reason": "",
      "reject": false
    }
  ]
}
```

## 7. Updated Scoring Formula

```
final_score =
  0.45 × image_similarity
+ 0.35 × semantic_similarity
+ 0.10 × category_match
+ 0.05 × tag_overlap
+ 0.05 × recency_or_usage_score
```

### Penalties

```
- 0.30  if query.primary_category = "Food"
        AND candidate.primary_category IN ("Research", "Technology", "Event")

- 0.25  if candidate.primary_category IN query.excluded_categories
```

### When to Trigger Claude Reranking

```
if top_score < 0.80
OR top 5 results are too close in score (delta < 0.03)
OR user explicitly clicks "More Accurate Suggestion"
→ run Claude reranker on top 10–20 only
```

## 8. Reclassification Job for Old Assets

Old assets with wrong embeddings will **not fix themselves**. Add a reclassification endpoint.

### Single Asset

```
POST /api/v1/media-assets/{id}/reclassify
```

### Batch (Admin)

```
POST /api/v1/media-assets/reclassify

{
  "scope": "institution",
  "institutionId": "uuid",
  "onlyLowConfidence": true,
  "confidenceThreshold": 0.70,
  "onlyBefore": "2026-05-28"
}
```

### Processing Steps

```
Load asset
↓
Fetch image from storage_url
↓
Run new improved Claude prompt
↓
Replace AI metadata fields
↓
Generate new IMAGE embedding via Voyage
↓
Generate new SEMANTIC embedding via Voyage
↓
Upsert rows in media_asset_embeddings
↓
Set reclassified_at = now()
```

Use one row per `(asset_id, embedding_type)`:

```sql
INSERT INTO media_asset_embeddings (asset_id, embedding_type, embedding, model)
VALUES (:assetId, :embeddingType, :embedding, :model)
ON CONFLICT (asset_id, embedding_type)
DO UPDATE SET
  embedding = EXCLUDED.embedding,
  model = EXCLUDED.model,
  created_at = now();
```

> **Recommended:** Run the batch job filtered to `onlyLowConfidence: true` first to fix the worst offenders without re-processing everything.

## 9. What NOT to Do

| ❌ Avoid | ✅ Do Instead |
|---|---|
| Claude scans every stored asset during suggestion | Claude scans only the query image once |
| Claude picks final results from entire library | pgvector retrieves top 30, Claude reranks top 10–20 |
| Single text embedding per asset | Image embedding + semantic embedding |
| Free-form Claude tags (random output) | Strict JSON schema with allowed category list |
| Skipping permission filter on vector search | Always filter by role before or during search |
| Returning soft-deleted or failed assets | Require `ma.status = 'READY' AND ma.deleted_at IS NULL` |

## 10. Migration Safety Notes

Do not drop the old `media_assets.embedding` column immediately. The current suggestion feature already depends on it, so migration must be staged.

Migration plan:

1. Create `media_asset_embeddings`.
2. Update new uploads to write both the old `media_assets.embedding` and the new `media_asset_embeddings` rows temporarily.
3. Backfill existing assets into the new table.
4. Backfill `media_assets.status` so existing embedded, non-deleted assets are `READY` and deleted assets are `DELETED`.
5. Update suggestion search to read from `media_asset_embeddings`.
6. Verify result quality, RLS behavior, and role filtering.
7. Only then remove or ignore the old `media_assets.embedding` column.

Flyway safety:
- Add this as a new migration version only. Do not edit an already-applied migration.
- If a shared Supabase database already has a higher migration version, use the next available version number.
- Keep `ON DELETE CASCADE` on child tables so hard deletes are deterministic.
- Keep the migration idempotent where possible with `IF NOT EXISTS` for new indexes and columns.
- Do not re-add `deleted_at`; it already exists as `TIMESTAMPTZ`.

## 12. Embedding Cleanup Requirement
When a media asset is deleted from the Media Library, all Voyage embeddings related to that asset must also be removed automatically or transactionally.
Do not leave orphaned vectors in the database.
Refactor the current single `media_assets.embedding` design into a separate `media_asset_embeddings` table:

For soft delete:
- Delete rows from `media_asset_embeddings` manually in the same transaction.
- Remove `asset_tags` rows or hide them with the parent asset.
- Set `media_assets.status = 'DELETED'`.
- Set `media_assets.deleted_at = now()`.
- Delete the Supabase file only if product policy says soft-deleted assets should not be recoverable; otherwise defer storage deletion to retention purge.
- Ensure every media library and suggestion query excludes deleted assets with `ma.status = 'READY' AND ma.deleted_at IS NULL`.

For hard delete:
- Delete the `media_assets` row.
- `media_asset_embeddings` are removed by `ON DELETE CASCADE`.
- `asset_tags` should also use `ON DELETE CASCADE`.
- Delete the Supabase storage object best-effort if it still exists.

| Upload (per asset) | 1 | 2 (image + semantic) | Fixed cost per upload |
